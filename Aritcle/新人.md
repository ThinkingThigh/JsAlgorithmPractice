# 前端新人快速上手指南

## 👋 欢迎加入团队

恭喜你加入我们的前端开发团队！这份指南将帮助你快速了解团队的工作方式、开发规范和技术栈，让你能够更快地上手并融入团队。

## 🔧 环境搭建

### 1. 开发环境要求
- **操作系统**: macOS/Windows/Linux
- **Node.js**: v16.0.0 或更高版本
- **npm/yarn**: npm 8.0+ 或 yarn 1.22+
- **Git**: 2.30.0+

### 2. 必备工具安装
```bash
# 安装 Node.js (推荐使用 nvm 管理版本)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.0/install.sh | bash
nvm install 18
nvm use 18

# 安装 Yarn (可选，但推荐)
npm install -g yarn

# 验证安装
node --version
npm --version
yarn --version
```

### 3. IDE 配置
推荐使用 VS Code，并安装以下插件：
- ESLint
- Prettier
- TypeScript Importer
- Auto Rename Tag
- Bracket Pair Colorizer
- GitLens
- Live Server

### 4. 项目克隆和初始化
```bash
# 克隆项目
git clone [项目仓库地址]
cd [项目目录]

# 安装依赖
yarn install  # 或 npm install

# 启动开发服务器
yarn dev     # 或 npm run dev
```

## 📁 项目结构了解

### 典型的前端项目结构
```
src/
├── components/     # 可复用组件
├── pages/         # 页面组件
├── hooks/         # 自定义 hooks
├── utils/         # 工具函数
├── services/      # API 服务
├── styles/        # 样式文件
├── types/         # TypeScript 类型定义
└── constants/     # 常量定义
```

### 重要文件说明
- `package.json`: 项目依赖和脚本
- `tsconfig.json`: TypeScript 配置
- `.eslintrc.js`: ESLint 配置
- `.prettierrc`: Prettier 配置
- `README.md`: 项目说明文档

## 📋 开发规范

### 1. 代码规范
- 使用 TypeScript 进行开发
- 遵循 ESLint 和 Prettier 配置
- 组件命名使用 PascalCase
- 文件命名使用 kebab-case
- 函数命名使用 camelCase

### 2. Git 提交规范
```bash
# 提交格式
<type>(<scope>): <subject>

# 常见类型
feat: 新功能
fix: 修复bug
docs: 文档更新
style: 代码格式调整
refactor: 代码重构
test: 测试相关
chore: 构建工具或辅助工具的变动
```

### 3. 分支管理
- `main/master`: 主分支，生产环境代码
- `develop`: 开发分支
- `feature/*`: 功能分支
- `bugfix/*`: 修复分支
- `release/*`: 发布分支

## 🔄 团队协作流程

### 1. 开发流程
1. 从 `develop` 分支创建功能分支
2. 在功能分支上进行开发
3. 提交代码前进行自测
4. 创建 Pull Request
5. 等待 Code Review
6. 合并到主分支

### 2. Code Review 规范
- 至少一人 Review 通过
- 解决所有评论
- 通过 CI/CD 检查
- 保持代码质量

## ❓ 常见问题解答

### 环境问题
**Q: 安装依赖时遇到权限问题？**
A: 使用 `sudo` 或修改 npm 全局安装路径

**Q: Node.js 版本不兼容？**
A: 使用 nvm 管理多个 Node.js 版本

### 开发问题
**Q: 如何调试代码？**
A: 使用浏览器开发者工具，或 VS Code 的调试功能

**Q: 代码格式化不一致？**
A: 确保安装了 Prettier 插件，并配置了项目格式化规则

### 协作问题
**Q: 如何处理 Git 冲突？**
A: 1. 保持冷静 2. 理解冲突原因 3. 手动解决冲突 4. 测试后提交

**Q: Code Review 被拒绝怎么办？**
A: 查看反馈意见，理解问题所在，修改后重新提交
